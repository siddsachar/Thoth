# Thoth Buddy Implementation Plan

This document defines the implementation-ready plan for **Buddy**, Thoth's prompt-generated animated companion. Buddy replaces the current sidebar avatar concept with a cute, useful, Thoth-branded companion that can live in the sidebar, float inside the app, and run as a desktop overlay.

## Finalized Product Decisions

- **Name:** Buddy
- **Primary runtime:** prompt-generated Buddy art plus generated image-to-video motion rendered by Thoth's canvas runtime
- **Default style:** cute mystical micro-familiar, Thoth-branded, expressive, warm, and not childish
- **Surfaces:** sidebar, in-app floating overlay, and native desktop overlay
- **Behavior model:** persistent local personality with mood, energy, focus, and alertness
- **Supported Buddies:** ship five bundled, curated, QA'd generated-video Buddy variants as the supported first-party set
- **Custom creation:** in-app Hatch flow for generated Buddy looks, state motion, and personality, using the user's configured media providers as a best-effort path
- **Settings ownership:** the Buddy tab owns Buddy identity, personality, surfaces, Hatch, collection, and overlay controls
- **Desktop ambition:** native overlay with platform-specific refinements after the pywebview path is working
- **Fallback posture:** if no Buddy asset exists yet, Buddy shows a clear bundled-look picker and Hatch action, then renders local still art until generated motion is available.
- **Asset posture:** bundled Buddies and Hatch Buddies use generated still art plus generated state-specific image-to-video clips. Rive is internal/future only and must not be exposed in the normal UI.

## Supported Asset Contract

Buddy has two product tiers:

- **Supported bundled Buddies:** Thoth ships five first-party Buddy variants generated by maintainers with the strongest available image and image-to-video models, manually reviewed for identity consistency, readable scale, loop quality, and state coverage. These are the supported, QA'd choices users can rely on.
- **Custom Hatch Buddies:** users can describe and generate their own Buddy inside Thoth. Hatch uses whatever image and video providers the user has configured. Output quality varies by model, so custom generation is explicitly best effort.

The supported asset workflow is:

1. Generate one canonical square Buddy image for the variant.
2. Use that exact image as the first frame for each state clip.
3. Generate six compact clips: idle, thinking, working, approval, success, and error.
4. Review clips as one identity set, rejecting variants where the character, silhouette, scale, or background drifts between states.
5. Bundle the approved preview image, clips, manifest, and metadata under `static/buddy/builtins/<buddy_id>/`.

Normal playback is local-only. Model calls happen only during Hatch generation or maintainer asset production, never during ordinary Buddy rendering.

## Product Goal

Buddy should feel like a small companion that understands what Thoth is doing without becoming noisy. It should communicate live state, ask for attention when needed, celebrate completions, and let the user jump directly into the relevant workflow.

Buddy is not just a decorative mascot. It is a compact status, notification, and interaction layer for Thoth.

## User-Facing Behavior

Buddy reacts to:

- Chat generation start, thinking, summarizing, tool calls, tool results, completion, interruption, and errors
- Workflow start, step progress, approval waits, completion, and failure
- Voice listening and TTS speaking
- Channel activity and approval responses
- Notifications, reminders, delivery failures, and available updates
- Long-running detached work such as Designer or browser tasks

Buddy can:

- Show compact bubbles with useful text
- Open the active thread
- Focus the chat input
- Open the Workflow Console
- Open approval controls
- Show the latest error or blocked state
- Sleep and wake
- Dock in the sidebar, drag into the workspace, and use a separate desktop overlay where native support exists
- Launch customization and hatch flows

## Implementation Architecture

### 1. Buddy Core Package


#### `buddy/events.py`

Owns the typed event bus.

Core types:

```python
class BuddyEventType(str, Enum):
    GENERATION_STARTED = "generation_started"
    GENERATION_THINKING = "generation_thinking"
    GENERATION_TOKEN = "generation_token"
    GENERATION_DONE = "generation_done"
    GENERATION_ERROR = "generation_error"
    TOOL_STARTED = "tool_started"
    TOOL_DONE = "tool_done"
    TOOL_ERROR = "tool_error"
    WORKFLOW_STARTED = "workflow_started"
    WORKFLOW_STEP = "workflow_step"
    WORKFLOW_DONE = "workflow_done"
    WORKFLOW_ERROR = "workflow_error"
    APPROVAL_REQUIRED = "approval_required"
    APPROVAL_RESOLVED = "approval_resolved"
    VOICE_LISTENING = "voice_listening"
    TTS_STARTED = "tts_started"
    TTS_DONE = "tts_done"
    CHANNEL_ACTIVITY = "channel_activity"
    NOTIFICATION = "notification"
    UPDATE_AVAILABLE = "update_available"
    USER_INTERACTION = "user_interaction"
```

Core dataclass:

```python
@dataclass
class BuddyEvent:
    type: BuddyEventType
    source: str
    message: str = ""
    thread_id: str | None = None
    task_id: str | None = None
    priority: int = 0
    payload: dict[str, Any] = field(default_factory=dict)
    created_at: float = field(default_factory=time.time)
    id: int = 0
```

Required helpers:

- `emit(event: BuddyEvent) -> None`
- `drain_events(since_id: int | None = None) -> list[BuddyEvent]`
- `latest_snapshot() -> list[BuddyEvent]`
- bounded in-memory queue to avoid unbounded growth
- monotonic event ids assigned by the event bus

#### `buddy/state.py`

Owns resolved Buddy state and personality persistence.

Core modes:

```python
class BuddyMode(str, Enum):
    IDLE = "idle"
    THINKING = "thinking"
    WORKING = "working"
    SPEAKING = "speaking"
    LISTENING = "listening"
    WAITING = "waiting"
    APPROVAL = "approval"
    SUCCESS = "success"
    ERROR = "error"
    SLEEPING = "sleeping"
```

Resolved state fields:

- `mode`
- `mood`: 0-100
- `energy`: 0-100
- `focus`: 0-100
- `alert`: 0-100
- `progress`: 0-100 or `None`
- `bubble`: optional short text
- `active_thread_id`
- `active_task_id`
- `trigger`: optional one-shot animation trigger
- `updated_at`

#### `buddy/brain.py`

Owns deterministic behavior resolution.

Responsibilities:

- Convert recent Buddy events into one resolved state
- Apply event priority
- Apply cooldowns to prevent jitter
- Track mood, energy, focus, and alert decay
- Decide short bubble text
- Emit one-shot animation triggers

Priority order:

```text
error > approval > waiting/user-input-needed > active tool > generation/thinking > speaking > listening > workflow > notification > idle
```

Behavior rules:

- Approval remains visible until resolved
- Error shows a stronger reaction briefly, then settles into a calm warning state
- Success reactions are short and should not interrupt new urgent states
- Token streaming should not trigger one animation per token
- Long-running work slowly lowers energy and raises focus
- Idle mode should use subtle ambient state, not chatter

#### `buddy/assets.py`

Owns Buddy pack discovery and validation. Users do not install or create Rive files. Generated Buddy art and generated motion packs are the normal live assets; any future rigged runtime remains internal to first-party asset production.

Built-in supported Buddy packs live under:

```text
static/buddy/builtins/<buddy_id>/
```

User-generated Buddy packs live under:

```text
~/.thoth/buddy/packs/<buddy_id>/
```

Generated Buddy motion-pack manifest format:

```json
{
  "schema": 1,
  "id": "lumen",
  "name": "Lumen",
  "runtime": "generated_motion_pack",
  "preview": "preview.png",
  "default_clip": "idle",
  "animation_map": {
    "idle_breathe": "idle",
    "think_loop": "thinking",
    "tool_peek": "working",
    "tap_glass": "approval",
    "pause": "error",
    "celebrate_small": "success",
    "worry": "error"
  },
  "clips": {
    "idle": { "path": "idle.mp4", "duration_seconds": 4 },
    "thinking": { "path": "thinking.mp4", "duration_seconds": 4 },
    "working": { "path": "working.mp4", "duration_seconds": 4 },
    "approval": { "path": "approval.mp4", "duration_seconds": 4 },
    "success": { "path": "success.mp4", "duration_seconds": 4 },
    "error": { "path": "error.mp4", "duration_seconds": 4 }
  }
}
```

Validation must ensure:

- Required manifest fields exist
- Runtime is `generated_motion_pack`
- Preview image exists and is displayable
- All referenced clip files exist and are non-empty
- Required state clips are present for idle, thinking, working, approval, success, and error
- Event-to-clip semantics stay explicit: `tap_glass` is approval-wait, `pause` is error/recovery for denied, timed-out, interrupted, stopped, or cancelled states, and only `celebrate_*` maps to success
- Built-in pack can be loaded from packaged app paths
- User pack paths cannot escape the Buddy data directory

Generated user pack manifest format:

```json
{
  "schema": 1,
  "id": "hatch_20260505_101500",
  "name": "Mira",
  "base_rig": "glyph",
  "preview": "preview.png",
  "prompt": "tiny moonlit study companion with gold eyes",
  "style": {
    "body_variant": "rounded",
    "accent_variant": "halo",
    "palette": {
      "body": "#26313f",
      "accent": "#e4c25e",
      "glow": "#4db8ab"
    }
  },
  "personality": {
    "preset": "warm_mystical",
    "description": "Curious, encouraging, calm under pressure"
  }
}
```

Generated pack validation must ensure:

- `base_rig` points to a valid shipped rig
- Preview exists and is displayable
- State motion clips exist when the pack claims animated support
- Palette values are valid colors
- Personality text is sanitized with the same safety posture as assistant identity text
- Pack paths cannot escape the Buddy data directory

#### `buddy/hatch.py`

Owns the in-app Hatch experience for creating a Buddy look and personality from a short description.

Initial Hatch pipeline:

1. Accept a short prompt and optional style chips
2. Generate 3 preview concepts using the existing image generation provider path
3. Convert the chosen concept into a template-compatible style recipe
4. Store preview image, prompt, style recipe, name, and personality locally
5. Produce a generated Buddy pack manifest
6. Activate the generated art through the procedural canvas runtime

Important boundary: custom Hatch generation is best effort. The supported experience is the bundled, QA'd first-party Buddy collection; Hatch uses the user's configured providers and should explain that model choice affects consistency.

#### `buddy/desktop.py`

Owns native overlay intent and lightweight state.

Responsibilities:

- Determine whether desktop mode is supported
- Store requested overlay mode and geometry
- Provide helpers used by `launcher.py` and UI actions

### 2. Prompt-Generated Runtime Assets

Important source-of-truth note: Buddy's v1 end-to-end product path is generated media, not a user-provided animation file. Hatch generates character art from the user's prompt, generates state-specific image-to-video clips from that same image, stores the preview plus motion-pack manifest as served Buddy assets, and the browser runtime switches clips on canvas with live mood, energy, focus, alert, and animation effects layered in.

The v1 product flow is:

1. User describes Buddy in Settings -> Buddy -> Hatch.
2. Thoth generates a single character image through the existing image generation provider.
3. Thoth sends that generated character image through the existing image-to-video provider path to create six state clips: idle, thinking, working, approval, success, and error.
4. Thoth saves the original preview, generated clips, and manifest under the Hatch draft, then copies the active versions to the served Buddy static directory as `current.png`, `current.mp4`, and `motions/manifest.json`.
5. Buddy surfaces pass the active generated image URL through `data-preview`, the default generated motion URL through `data-motion`, and the state motion manifest through `data-motion-pack`.
6. `static/buddy/runtime/buddy.js` preloads the generated video clips, switches clips by resolved Buddy animation, and falls back to the still image when needed.

The current generated-motion implementation uses image-to-video first because it reuses Thoth's existing media provider path. A later richer version can add transparent video, spritesheets, or richer clip metadata, but it must preserve the supported bundled-pack contract and keep normal playback local-only.

Add:

```text
static/buddy/runtime/buddy.js
static/buddy/builtins/glyph/manifest.json
static/buddy/builtins/glyph/preview.png
static/buddy/builtins/glyph/motions/idle.mp4
static/buddy/builtins/glyph/motions/thinking.mp4
static/buddy/builtins/glyph/motions/working.mp4
static/buddy/builtins/glyph/motions/approval.mp4
static/buddy/builtins/glyph/motions/success.mp4
static/buddy/builtins/glyph/motions/error.mp4
~/.thoth/buddy/generated/current.png
~/.thoth/buddy/generated/current.mp4
~/.thoth/buddy/generated/motions/manifest.json
```

`buddy.js` responsibilities:

- Mount a canvas into each Buddy target element
- Load the active generated Buddy motion pack from `data-motion-pack`
- Fall back to `data-motion` and then `data-preview` when a pack or clip is unavailable
- Render generated motion on canvas without moving the source background square
- Apply resolved state updates
- Fire one-shot triggers once
- Render compact speech/status bubbles
- Support draggable floating mode
- Report runtime load errors clearly

Global API:

```javascript
window.thothBuddy = {
  mount(targetId, options),
  unmount(targetId),
  setState(state),
  trigger(name),
  showBubble(text, options),
  hideBubble(),
  setSurface(surface),
  setInteractionHandler(handlerName)
};
```

  - `on_top=True`

Platform-specific notes:

- macOS: pursue transparent/frameless overlay behavior after basic route works
- Windows: pywebview transparency is limited; SOTA path may require Win32 layered-window refinements
- Linux: desktop overlay is optional and dependent on desktop environment support; browser-first remains supported

Windows first-show transparency fix, 2026-05-06: the desktop overlay now creates the primary pywebview Buddy window with `hidden=True` and sets `_BUDDY_WINDOW_READY=False` until `/buddy-overlay` explicitly calls `mark_buddy_window_ready()` after transparent document/body/NiceGUI/Quasar styles are applied and the Buddy root has rendered. Non-manual `show_buddy_window(false)` calls, including main-window focus/blur sync, are ignored until this ready marker so the hidden WebView cannot be revealed during its default white first paint. Fallback create-window candidates intentionally drop `hidden` and then transparency hints so older or stricter pywebview backends can still open the overlay visibly. Do not use `background_color="#00000000"` as a quick fix; that alpha hint broke Buddy window creation on Windows.

### 9. Supported Buddy Art Direction

Default built-in Buddy: `glyph`. Current bundled generated-motion packs are `glyph`, `lumen`, `ember`, `pixel`, `sprout`, and `orbit`. Keep first-party variants intentionally varied so Buddy does not read as a single Thoth-branded palette.

Variant targets:

- `glyph`: mystical ink-and-gold micro-familiar
- `lumen`: luminous archive guide with moonlit ink, warm gold, teal glass accents, and a folded-page collar
- `ember`: forge-spark task helper with charcoal ink, copper-gold glow, teal circuit accents, and a practical tool posture
- `pixel`: retro arcade companion with a translucent CRT shell, cyan/magenta edge lights, yellow pixel eyes, and a joystick charm
- `sprout`: botanical greenhouse companion with a seedpod hood, dewdrop lantern core, tiny garden, and gardener tools
- `orbit`: space mission-control companion with a pearl micro spacesuit, glossy visor, blue/amber status lights, and star-map panel

Direction:

- Small compact companion
- Soft silhouette with readable eyes
- User concept owns the theme, palette, materials, and era; avoid forcing ancient, mystical, ink/gold/teal, glyph, or Thoth-like motifs unless requested
- Expressive posture and eye animation
- Calm idle breathing
- Focused thinking lean
- Tool-work loop
- Approval attention gesture
- Success bounce
- Error concern reaction
- Sleep curl/rest pose

Avoid:

- Overly corporate assistant orb
- Generic pixel-style desktop companion clone
- Literal deity depiction
- Distracting constant motion
- Large speech bubbles that cover app content

### 10. Hatch Flow UX

Hatch dialog steps:

1. Name your Buddy, or keep the current Buddy name
2. Describe your Buddy in 1-3 lines
3. Choose simple vibe chips: cozy, clever, scholarly, cosmic, tiny guardian, playful, calm
4. Generate 3 preview cards
5. Pick one result or refine the prompt
6. Save the Buddy look
7. Activate it now or keep it in the collection

Implementation boundary:

- The first implementation can create preview cards, style recipes, generated pack metadata, and local collection entries
- Activation uses generated Buddy art, generated motion, and the canvas runtime
- Custom output is best effort because it depends on the user's configured image and video providers

## Implementation Order

### Wave 1: Buddy Core Contract

Deliverables:

- `buddy` package skeleton
- Event bus
- Resolved state model
- Brain priority/cooldown logic
- Config read/write
- Asset manifest validation
- Unit tests

Acceptance criteria:

- Events can be emitted and drained safely
- Brain resolves correct priority order
- Mood/energy/focus/alert decay is deterministic
- Config is created with defaults
- Built-in manifest validation works

### Wave 2: Generated Runtime and Sidebar Surface

Deliverables:

- Generated canvas runtime loader
- `buddy.js`
- `ui/buddy.py`
- Sidebar Buddy mount
- Default `glyph` manifest

Acceptance criteria:

- Sidebar loads Buddy canvas
- Buddy receives resolved state updates
- Generated canvas animation updates from Python state
- Runtime error is visible and non-fatal

### Wave 3: Event Instrumentation

Deliverables:

- Streaming events
- Tool call events
- Workflow events
- Approval events
- Notification events

Acceptance criteria:

- Buddy reacts to real chat generation
- Buddy reacts to tool start/done
- Buddy reacts to approval-required state
- Buddy reacts to workflow running/done/error

### Wave 4: Dockable In-App Buddy

Deliverables:

- One in-app Buddy that starts in the sidebar dock and can be dragged into the workspace
- Empty sidebar dock state while Buddy is away
- In-app and desktop surface settings
- Contextual click behavior
